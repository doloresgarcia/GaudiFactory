# Phase 5: Documentation

> Read `methodology/03-phases.md` → "Phase 5" for full requirements.

**Goal:** Self-contained, usable documentation so another developer can
understand, build, and use the algorithm without reading the source.

## Tasks

### 1. Doxygen-style comments in header

Add to `src/components/MyAlg.h`:

```cpp
/**
 * @class MyAlg
 * @brief Brief one-line description.
 *
 * Longer description: what this algorithm does, the Gaudi pattern used,
 * and a note on when to use it vs. alternatives.
 *
 * Pattern: k4FWCore::Producer / Consumer / Transformer / Gaudi::Algorithm
 *
 * @author Generated by GaudiFactory
 */
class MyAlg : public ... {
public:
  /// Constructor
  MyAlg(const std::string& name, ISvcLocator* svcLoc);

  /// Brief description of execute/operator()
  StatusCode execute() override;

private:
  /// Input collection name. Default: "MCParticles"
  Gaudi::Property<std::string> m_inputName{...};

  /// Energy threshold in GeV. Particles below this are discarded.
  Gaudi::Property<float> m_minEnergy{...};
};
```

Every property must have an inline comment describing its effect.

### 3. Update `DESIGN.md`

Add a final section "Implementation notes" to `phase1_design/outputs/DESIGN.md`
noting any deviations from the original design and why.

## Self-check before review

- [ ] Every property has an inline doxygen comment
- [ ] Class-level doxygen comment present with pattern justification
- [ ] README covers: purpose, pattern, collections, properties, build, run, example output
- [ ] README references at least one figure from Phase 4
- [ ] No "TODO" left in comments

## Review

2-bot: critical + constructive reviewers.
- Critical: checks completeness — every property documented, all
  collections in README table, build instructions tested.
- Constructive: clarity and usability — would a new developer understand
  how to use this algorithm from the README alone?
